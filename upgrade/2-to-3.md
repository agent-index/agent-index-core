# Upgrade: agent-index-core v2.x → v3.0.0

## Summary

v3.0.0 replaces the persistent MCP server and bridge daemon with an on-demand executor for all remote filesystem operations. The `aifs_*` tool interface is unchanged — collections and member workflows continue to use the same tool names and arguments. The change is in *how* those tools are invoked: instead of routing through a running MCP server process, Claude calls a shell wrapper (`aifs-exec.sh`) directly via bash. Each call spawns a fresh Node process, executes one operation, and exits.

This eliminates the class of failures where Cowork's platform-level process management terminated the MCP server mid-session, and removes the bridge daemon that was added as a workaround.

**What members experience:** After applying the update, everything works the same way — `aifs_read`, `aifs_write`, and all other `aifs_*` tools behave identically. Existing authentication credentials are preserved. The only visible change is that the filesystem plugin validates the exec bundle at session start instead of starting a server process.

---

## Breaking Changes

1. **MCP server removed.** The persistent `server.bundle.js` process is replaced by on-demand `aifs-exec.bundle.js`. Claude no longer sends tool calls to an MCP server — it calls the shell wrapper via bash.
2. **Bridge daemon removed.** The HTTP bridge (`aifs-bridge.mjs`) that auto-restarted the MCP server is no longer needed or present.
3. **`agent-index.json` schema changed.** The `remote_filesystem.mcp_server` key is replaced by `remote_filesystem.exec`. See config migration below.
4. **Cowork plugin updated.** The plugin no longer starts a server process. It validates the exec bundle and exits. Members must install the updated plugin (included in the new bootstrap zip).
5. **Bootstrap zip contents changed.** The zip now includes `aifs-exec.bundle.js` and `aifs-exec.sh` instead of `server.bundle.js`. The directory structure under `mcp-servers/filesystem/` has changed.

---

## What Does NOT Change

- **`aifs_*` tool interface** — same tool names, same arguments, same return types, same error codes.
- **Authentication model** — per-member credentials stored at `.agent-index/credentials/` are preserved. No re-authentication required.
- **Remote filesystem layout** — all remote paths, directory structure, and shared files are unchanged.
- **Collection definitions** — all skill and task definitions work without modification (they reference `aifs_*` tools, which still exist).
- **Member workspaces** — all local member files are preserved.
- **`session-bootstrap.sh`** — the hook script performs local checks only and is unchanged in function.

## Config Migration

### `agent-index.json` — `remote_filesystem` section

**v2.x (old):**

```json
{
  "remote_filesystem": {
    "backend": "gdrive",
    "mcp_server": {
      "adapter": "gdrive",
      "adapter_version": "1.0.0",
      "bundle_path": "mcp-servers/filesystem/server.bundle.js"
    },
    "auth": {
      "method": "per-member",
      "credential_store": ".agent-index/credentials/"
    },
    "connection": { ... }
  }
}
```

**v3.0.0 (new):**

```json
{
  "remote_filesystem": {
    "backend": "gdrive",
    "exec": {
      "adapter": "gdrive",
      "adapter_version": "2.0.0",
      "bundle_path": "mcp-servers/filesystem/aifs-exec.bundle.js",
      "shell_wrapper": "mcp-servers/filesystem/aifs-exec.sh"
    },
    "auth": {
      "method": "per-member",
      "credential_store": ".agent-index/credentials/"
    },
    "connection": { ... }
  }
}
```

Changes:
- `mcp_server` key → `exec` key
- `adapter_version` bumped to `2.0.0` (adapter repos also migrated to exec)
- `bundle_path` points to `aifs-exec.bundle.js` instead of `server.bundle.js`
- New `shell_wrapper` field added, pointing to `aifs-exec.sh`
- `auth` and `connection` sections are unchanged

## Migration Steps

### Step 1 — Admin: Generate New Bootstrap Zip

Run `@ai:create-org` with the `--regenerate-bootstrap` flag (or re-run the bootstrap zip generation step). This produces a new zip containing:

```
mcp-servers/
  filesystem/
    aifs-exec.bundle.js    ← new: on-demand exec bundle
    aifs-exec.sh           ← new: shell wrapper
    adapter.json           ← updated adapter metadata
agent-index.json           ← updated with exec config
agent-index-core/
  .claude/hooks/
    session-bootstrap.sh   ← v3.0.0 bootstrap script
CLAUDE.md                  ← updated context
.claude/
  settings.json            ← SessionStart hook (unchanged)
```

The old `server.bundle.js` is NOT included. The new zip is self-contained — members who unpack it get a clean v3.0.0 install.

Upload the new bootstrap zip to the remote filesystem at `/shared/bootstrap/member-bootstrap.zip`, replacing the v2.x zip.

### Step 2 — Admin: Publish Update Instructions

Run `@ai:publish-updates` to create a `core-update` entry in the update log targeting v3.0.0. The update instructions should include the member migration steps below.

### Step 3 — Members: Apply Update

Each member runs `@ai:apply-updates`, which performs the following. If a member prefers to apply manually, the steps are listed here:

#### 3a. Remove Old Server and Bridge Files

Delete the old MCP server bundle, bridge daemon, and bridge tools from the local install:

```
mcp-servers/filesystem/server.bundle.js     ← delete
mcp-servers/filesystem/aifs-bridge.mjs      ← delete (if present)
mcp-servers/filesystem/bridge.log           ← delete (if present)
agent-index-core/tools/aifs-bridge/         ← delete entire directory (if present)
```

The `agent-index-core/tools/aifs-bridge/` directory contains `aifs-bridge.mjs` and `aifs-call.sh` from v2. These reference `server.bundle.js` which no longer exists and will cause errors if Claude discovers and tries to use them. The `mcp-servers/filesystem/` directory is reused for the new exec files.

#### 3b. Install New Exec Files

Copy the new exec bundle and shell wrapper into the local install:

```
mcp-servers/filesystem/aifs-exec.bundle.js  ← new
mcp-servers/filesystem/aifs-exec.sh         ← new
mcp-servers/filesystem/adapter.json         ← updated
```

The shell wrapper must be executable: `chmod +x mcp-servers/filesystem/aifs-exec.sh`.

#### 3c. Update `agent-index.json`

Replace the `remote_filesystem.mcp_server` key with `remote_filesystem.exec` as shown in the config migration section above. The `auth` and `connection` sections are unchanged.

#### 3d. Update Cowork Plugin (Cowork users only)

If the member uses Cowork, install the updated `agent-index-filesystem.plugin` from the new bootstrap zip. The updated plugin validates the exec bundle at session start instead of starting an MCP server process. Open the `.plugin` file and confirm the install prompt.

CLI users (Claude Code) do not need a plugin — the session-bootstrap hook and session-start task handle everything.

#### 3e. Update Core Files

Replace the following files from the new bootstrap zip or the update package:

- `agent-index-core/.claude/hooks/session-bootstrap.sh` — v3.0.0 bootstrap script
- `CLAUDE.md` — updated context with exec-model references

### Step 4 — Members: Verify

After applying the update, verify:

1. **Exec bundle exists:** `ls mcp-servers/filesystem/aifs-exec.bundle.js` — file should be present
2. **Shell wrapper is executable:** `bash mcp-servers/filesystem/aifs-exec.sh aifs_auth_status '{}'` — should return JSON with `authenticated: true`
3. **Config is correct:** Read `agent-index.json` and confirm the `remote_filesystem` section has an `exec` key (not `mcp_server`)
4. **Old files removed:** Confirm `server.bundle.js` and `aifs-bridge.mjs` no longer exist in `mcp-servers/filesystem/`, and `agent-index-core/tools/aifs-bridge/` directory no longer exists
5. **Remote access works:** `aifs_read("/org-config.json")` returns the org configuration

## Preserved Member Data

All member-local files are preserved:
- `member-index.json` and all installed skill/task state
- `profile/` directory (preferences, role config)
- `.agent-index/credentials/` — authentication tokens carry over unchanged
- `.agent-index/logs/` — existing logs preserved

---

## Automatic Re-authentication

Existing authentication credentials are preserved across the upgrade. The `aifs_*` tool interface is unchanged, so the same OAuth tokens, AWS SSO sessions, or other credentials continue to work. Members do NOT need to re-authenticate unless:

- Their OAuth refresh token has expired (Google testing apps: 7-day expiry)
- The admin revoked the OAuth app or rotated credentials
- The credential file was deleted during manual cleanup

If a member encounters `NOT_AUTHENTICATED` after upgrading, they can run `@ai:member-bootstrap` to re-authenticate.

---

## Rollback

If the upgrade fails for a member:

1. Restore `server.bundle.js` from a backup or the old bootstrap zip
2. Restore the `mcp_server` key in `agent-index.json` (replace `exec` with `mcp_server`, remove `shell_wrapper` field)
3. The old MCP server model will work with the v2.x core files

Note: rollback means the member cannot use v3.0.0 features or receive further updates until they complete the migration. Other members on v3.0.0 are unaffected — the remote filesystem is compatible with both versions.

---

## Notes

- The path cache file (`path-cache.json` in `mcp-servers/filesystem/`) is compatible across versions. If it exists from v2.x usage, the v3.0.0 exec bundle will use it.
- The `@ai:apply-updates` flow handles all of Steps 3a–3e automatically. Manual steps are documented for transparency and for members who prefer manual control.
- Collections that declare `agent_index_min_version: "3.0.0"` will not install on a v2.x core. Members must complete the core upgrade before installing or updating collections that require v3.0.0.
