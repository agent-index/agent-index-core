# Upgrade: agent-index-core v1.x → v2.0.0

## Summary

> **Note:** This guide covers the v1→v2 migration. v3.0.0 subsequently replaced the MCP server with an on-demand executor — see the 2-to-3 upgrade guide.

v2.0.0 replaces the shared-mount-drive filesystem model with a remote filesystem model. Org/shared files now live on a remote storage backend (Google Drive, OneDrive, or S3) accessed via the `aifs_*` remote filesystem tools. Member files remain local only.

---

## Breaking Changes

1. **Filesystem model**: All org-level reads/writes now use `aifs_*` MCP tools instead of direct filesystem access.
2. **`filesystem-setup` removed**: Replaced by `member-bootstrap` skill, which handles remote authentication, connectivity verification, local workspace creation, and remote member registration.
3. **`session-start` updated**: Now checks `aifs_auth_status()` instead of local filesystem mount.
4. **`create-org` rewritten**: Uploads org files to remote, generates bootstrap zip for member distribution.
5. **`agent-index.json` schema changed**: New `remote_filesystem` section and `local.members_root` field.
6. **`org-config.json` schema changed**: New `remote_filesystem` and `paths` sections.

---

## Migration Steps

### Step 1 — Admin: Set Up Remote Filesystem

The org admin must:

1. Choose a remote storage backend (Google Drive, OneDrive, or S3)
2. Install the corresponding filesystem adapter (`agent-index-filesystem-gdrive`, `agent-index-filesystem-onedrive`, or `agent-index-filesystem-s3`)
3. Run `@ai:create-org` to initialize the remote filesystem with the org's existing configuration

### Step 2 — Admin: Migrate Org Data

1. Upload existing org files from the old shared mount to the remote filesystem using `aifs_write`
2. Verify org-config.json, members-registry.json, and collection directories are accessible via `aifs_read`
3. Generate the bootstrap zip via `@ai:create-org` for distribution to members

### Step 3 — Admin: Publish Update Instructions

Run `@ai:publish-updates` to create a `core-update` entry in the update log targeting v2.0.0.

### Step 4 — Members: Apply Update

Each member runs `@ai:apply-updates`, which will:

1. Install the filesystem adapter exec bundle
2. Run `@ai:member-bootstrap` to authenticate with the remote filesystem
3. Update local `agent-index.json` with the new `remote_filesystem` section
4. Verify connectivity via `aifs_auth_status()`

### Step 5 — Members: Verify

After applying the update, members should verify:

- `aifs_auth_status()` returns `authenticated: true`
- `aifs_read("/org-config.json")` returns the org configuration
- Local workspace at `members/{member_hash}/` is intact

---

## Preserved Member Data

All member-local files are preserved:
- `member-index.json`
- `skills/` directory and all installed skill state
- `tasks/` directory and all installed task state
- `profile/` directory (preferences, role config, onboarding state)

---

## Removed Capabilities

- `filesystem-setup` skill — replaced by `member-bootstrap`. Members who had `filesystem-setup` installed will have it automatically replaced during the update flow.
