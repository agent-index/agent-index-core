## About This Task

Upload-install-log sends install diagnostic logs (generated by `create-org` and `member-bootstrap`) to the agent-index development team's log collection endpoint. These logs help the team identify install problems, improve error handling, and prioritize fixes.

The logs contain only hashed identifiers (org hash, member hash), step-level diagnostics, and Claude's reasoning during the install process. No credentials, tokens, file contents, or personally identifiable information is included.

### Inputs

No required inputs. The task reads log files from `.agent-index/logs/` and the configuration from `agent-index.json`.

### Outputs

The log file is POSTed to the log collector endpoint. On success, the task surfaces a confirmation with the assigned log ID. The local log file is not deleted — it remains available for local review.

### Cadence & Triggers

Run manually by the admin or member, or offered at the end of `create-org` and `member-bootstrap`.

## Workflow

### Step 1: Check for Log Files

List files in `.agent-index/logs/` matching `*.jsonl`. If no log files exist: surface "No install logs found. Logs are generated automatically during org setup and member setup." Halt.

If multiple log files exist: show the list with timestamps and types (parsed from filenames), and ask which to upload. Offer "all" as an option.

If exactly one log file exists: use it.

### Step 2: Read Configuration

Read `agent-index.json` and extract:
- `log_collector_url` — the endpoint URL
- `log_collector_api_key` — the API key

If either is missing: surface "Log upload is not configured. The `log_collector_url` and `log_collector_api_key` fields in `agent-index.json` need to be set." Halt.

### Step 3: Build the Envelope

For each log file being uploaded, construct the JSON envelope:

```json
{
  "schema_version": "1.0",
  "log_type": "{parsed from filename — 'create-org' or 'member-bootstrap'}",
  "run_id": "{parsed from filename}",
  "org_hash": "{computed from org name in org-config.json or install-state.json}",
  "member_hash": "{the current member's hash from session context}",
  "agent_index_version": "{version from agent-index.json}",
  "adapter_backend": "{backend from agent-index.json remote_filesystem section}",
  "submitted_at": "{current ISO 8601 timestamp}",
  "entries": [{the parsed JSONL entries as a JSON array}]
}
```

**Computing org_hash:** Read `org-config.json` from the remote filesystem via `aifs_read("/org-config.json")` if available, or from `.agent-index/install-state.json` if remote is not accessible. Compute SHA256 of the lowercase org name, prefix with `sha256:`, and truncate to 16 hex characters (same convention as member identity resolution).

**Computing member_hash:** Use the current member's hash from the session bootstrap context. If not available (e.g., running during an incomplete install), compute from the member's email using the same SHA256 convention.

**Handling missing fields:** If `org_hash` or `member_hash` cannot be determined, use `"unknown"`. The upload should still proceed — partial metadata is better than no log.

### Step 4: Upload

For each envelope, send an HTTP POST to `log_collector_url`:
- Header: `Authorization: Bearer {log_collector_api_key}`
- Header: `Content-Type: application/json`
- Body: the JSON envelope

Use a simple `curl` command or equivalent HTTP call from the shell.

**On success (HTTP 200):** Parse the response to get the `id` field. Surface: "Install log uploaded successfully (ID: {id})."

**On HTTP 401/403:** Surface: "The API key was rejected. Check that `log_collector_api_key` in `agent-index.json` is correct."

**On HTTP 413:** Surface: "The log file is too large to upload. This is unusual — install logs are typically small."

**On network error:** Surface: "Couldn't reach the log collection endpoint. Check network connectivity and that `log_collector_url` in `agent-index.json` is correct. You can try again later with '@ai:upload-install-log'."

**On other errors:** Surface the HTTP status and response body for debugging.

### Step 5: Confirm

After all uploads complete, summarize: "{N} log(s) uploaded. Thank you — this helps the agent-index team improve the install experience."

## Directives

### MCP Tool Usage

This task may use `aifs_read` to read `org-config.json` for the org name hash. If the MCP tools are not available (e.g., running before MCP server is configured), fall back to reading `.agent-index/install-state.json` locally. This is one of the few tasks where MCP tool unavailability is acceptable — the upload should work even during a partially completed install.

### Behavior

This task handles potentially sensitive diagnostic data. Before uploading, briefly explain what the log contains: "The install log contains step-by-step diagnostics from your setup process — which steps ran, timing, errors encountered, and Claude's reasoning. It includes hashed identifiers for your org and member but no credentials, tokens, or personal information."

Ask for confirmation: "Ready to upload?" Proceed only on explicit approval.

Keep the output minimal. The member doesn't need to see the full envelope — just the file being uploaded, the destination, and the result.

### Constraints

Never upload a log file without explicit user confirmation.

Never include credentials, tokens, or OAuth secrets in the envelope. The logging directives in `create-org` and `member-bootstrap` already prohibit logging these, but verify by scanning the entries for common patterns (`client_secret`, `access_token`, `refresh_token`, `Bearer`) before upload. If found, redact the values (replace with `[REDACTED]`) and surface a warning: "Some entries contained credential-like values which have been redacted before upload."
